# WebRTC & Streaming Security Checklist

Reference for auditing the monitor/WebRTC subsystem. Babything uses a custom signaling relay (API server) with P2P media between browser and agent.

## Architecture Recap

```
Browser ←──HTTP API (signaling)──→ API Server ←──WebSocket──→ Agent (camera bridge)
   ↓                                                            ↑
   └────────────── WebRTC P2P (DTLS-SRTP) ──────────────────────┘
                            ↓
                      TURN relay (fallback)
```

- API server never sees video/audio payload — only SDP offers/answers and ICE candidates.
- Agent is a Docker container running on the user's local hardware.
- TURN server (`coturn`) relays media when direct P2P fails.

## Checklist

### Agent Authentication

- [ ] Agent token is JWT with short expiry (not 30+ days)
- [ ] Token is scoped to `tenantId` + `scope: 'monitor'`
- [ ] Token is verified on every WebSocket connection
- [ ] Only one agent per tenant allowed; reconnection evicts old session
- [ ] Token rotation is possible without downtime
- [ ] Token is NOT logged, returned in API responses, or stored in client-side code

### Watch Session Authorization

- [ ] `POST /monitor/watch` requires authenticated user of the tenant
- [ ] Watch sessions are scoped to `tenantId`
- [ ] Cross-tenant watch session access returns 404 (not 403 — avoid information leak)
- [ ] Watch sessions auto-expire (currently 5 min inactivity prune)
- [ ] Watch sessions are destroyed on browser disconnect / page unload
- [ ] Only users with baby access (or admin) can initiate watch sessions

### Signaling Security

- [ ] SDP offers/answers do not contain internal IP addresses or network topology
- [ ] ICE candidates are filtered to remove sensitive host candidates if possible
- [ ] Long-polling endpoints have reasonable timeout and connection limits
- [ ] WebSocket has ping/pong keepalive
- [ ] Malformed signaling messages are rejected without crashing the server
- [ ] Signaling happens over HTTPS/WSS (not HTTP/WS in production)

### TURN / STUN Security

- [ ] TURN credentials are **not** shared long-term secrets visible to all users
- [ ] TURN credentials are ephemeral (time-limited, single-use, or rotated)
- [ ] TURN server requires authentication (`lt-cred-mech`)
- [ ] TURN server disables weak TLS (`--no-tlsv1`, `--no-tlsv1_1`)
- [ ] TURN server is not exposed to the public internet unnecessarily
- [ ] STUN/TURN config endpoint (`GET /monitor/config`) does not leak internal IPs

### Media Encryption

- [ ] WebRTC uses DTLS 1.2+ for key exchange
- [ ] SRTP is used for media payload encryption
- [ ] No downgrade to unencrypted RTP is possible
- [ ] Agent→camera (RTSP) is on local network only — verify no WAN exposure

### Session Lifecycle

- [ ] Active watch sessions count is bounded (prevent DoS)
- [ ] Agent disconnect immediately invalidates active watch sessions
- [ ] Browser tab close / navigation stops the stream and cleans up
- [ ] Watch session IDs are unguessable (CUID/UUID)
- [ ] No dangling watch sessions after API restart (in-memory state is expected, but document it)

### Agent Token UI Flow

- [ ] Admin UI for generating agent tokens requires re-authentication (sensitive action)
- [ ] Generated token is shown once, not stored in database or logs
- [ ] Token revocation UI exists
- [ ] Old tokens are invalidated when a new one is generated

### Rate Limiting & DoS

- [ ] Monitor endpoints have rate limits
- [ ] WebSocket connections per IP are bounded
- [ ] Watch session creation per user is bounded
- [ ] ICE candidate polling has backoff / limits

---

## Known Patterns in This Codebase

- Agent token: 30-day JWT generated by `POST /monitor/token` (admin-only)
- Watch sessions: in-memory Maps, pruned after 5 min inactivity
- TURN credentials: returned to any authenticated user via `GET /monitor/config`
- ICE servers: configured via env vars `TURN_URL`, `TURN_USERNAME`, `TURN_CREDENTIAL`
- Agent WS auth: query param `?token=<jwt>` — verify this is never logged by nginx or the app
